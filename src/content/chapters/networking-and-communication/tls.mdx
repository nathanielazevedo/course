TCP gives you a reliable byte stream. By itself it gives you nothing else — anyone with access to a router along the path can read the bytes, modify them, or impersonate either endpoint. TLS — Transport Layer Security — is the layer that fixes that. The `S` in HTTPS is TLS.

### What TLS Provides

TLS gives a TCP connection three things:

- **Confidentiality** — the bytes on the wire are encrypted; an observer sees ciphertext.
- **Integrity** — any modification in transit is detected; a tampered byte fails the message authentication check.
- **Authentication** — the client can verify it is actually talking to the server it intended (and optionally vice versa).

These properties together are called *encryption-in-transit*. TLS does not encrypt data at rest, does not hide *who* you are talking to (the IP and SNI hostname are visible), and does not protect a compromised endpoint from itself.

### Symmetric vs Asymmetric Cryptography

TLS uses both kinds of cryptography, for different reasons.

**Symmetric** encryption uses one shared key for both encrypting and decrypting. It is fast — modern CPUs have hardware instructions for AES — but it requires the two parties to already share a secret.

**Asymmetric** (public-key) encryption uses a key *pair*: a public key anyone can know, and a private key only the owner has. Things encrypted with the public key can be decrypted only with the private one. It is much slower than symmetric, but it solves the problem of agreeing on a secret over an untrusted network.

TLS uses asymmetric crypto only at the start, to safely agree on a symmetric key. The bulk of the data — every byte of the actual HTTP request and response — is encrypted with the cheap symmetric cipher.

### Certificates and PKI

A **certificate** is a public key plus identifying information (which domain it belongs to, who issued it, when it expires) plus a signature from a *Certificate Authority* attesting that the bound identity is real.

Your operating system ships with a list of root CAs it trusts. When a server presents a certificate, the client checks:

1. The certificate is signed by a CA the client trusts (or by a chain leading back to one).
2. The certificate has not expired and has not been revoked.
3. The certificate's listed domain name matches the host the client connected to.

If any check fails, the connection is aborted before any application data flows. This is why a wrong-name or self-signed certificate produces a browser warning.

The whole system is called **PKI** — public key infrastructure. Its security rests on the integrity of the CAs. A compromised CA can issue valid certificates for any domain. This has happened, and is why CAs are watched closely (Certificate Transparency logs every issued cert publicly so unauthorized issuance is detectable).

### The Handshake

A TLS 1.3 handshake — one round trip on top of the TCP handshake:

1. **Client → Server**: "ClientHello" — the TLS version, supported ciphers, a random number, and a Diffie-Hellman key share.
2. **Server → Client**: "ServerHello" — the chosen cipher, its own random number, its DH key share, its certificate, and a signature over the handshake so far. From this exchange both sides can independently derive the same symmetric session key.
3. **Client**: verifies the certificate chain, sends "Finished" encrypted with the new key.

After the handshake, both sides hold a shared secret derived through Diffie-Hellman. The session key is used with a symmetric cipher for the rest of the connection.

TLS 1.2 needed two round trips. TLS 1.3 needs one. **0-RTT** mode lets a client resume a previous session and send application data with the very first packet, at the cost of replay safety for that data.

### Forward Secrecy

The session key is derived freshly per connection from the Diffie-Hellman exchange — it is never sent over the wire and is discarded when the connection ends. Even if an attacker records all the encrypted traffic and later steals the server's private key, they cannot decrypt past sessions. This property is called **forward secrecy** and is the default in TLS 1.3.

### What TLS Does Not Do

- **Hide that you are talking to the server.** The server's IP address and the SNI hostname are unencrypted in the ClientHello. (Encrypted SNI / ECH is being deployed but not universal.)
- **Protect against a compromised endpoint.** If the server is malicious or breached, TLS faithfully delivers your data to it.
- **Authenticate the client by default.** Mutual TLS — where the client also presents a certificate — is supported but rare outside service-to-service traffic.
- **Protect data after it arrives.** TLS terminates at the load balancer, the application, or wherever it is configured to terminate. Anything past that point is protected by something else (or nothing).

### What This Means in Practice

Almost every external connection your code makes runs over TLS. The cost is real — extra round trips at connection time, CPU for encryption, certificates to manage — but it has become the floor, not a feature. An HTTP-only endpoint in 2026 is treated as broken by browsers, search engines, and increasingly by libraries.

That gives us TCP for reliable bytes, DNS for finding the other side, and TLS for trusting it. The next chapter is what runs on top: HTTP and the web.
