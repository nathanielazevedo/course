The CPU runs one instruction at a time, in a loop. Each iteration is one instruction cycle (also called the fetch–decode–execute cycle), and it runs billions of times per second.

### The Program Counter

The data the CPU is actively working on lives in registers, the top tier of the memory hierarchy. One register is special: the *program counter* (PC). It holds the memory address of the next instruction to run. Every iteration of the cycle is anchored on it.

### Fetch

Read the instruction at the address in the program counter. Load it into the CPU.

The program counter then advances to point at the next instruction.

### Decode

Determine what the instruction means. Each instruction is a fixed binary pattern that maps to a specific operation — *add these two numbers*, *load this memory address*, *jump to that one*.

### Execute

Carry out the operation. The CPU updates registers, reads or writes memory, or modifies the program counter.

A jump instruction changes the program counter to a new address. That is how loops, branches, and function calls work — without jumps, every program would run straight through to the end.

### The Clock

A signal that ticks at a fixed frequency, driving each step of the cycle. A 3 GHz clock ticks 3 billion times per second.

Clock speed is one measure of a CPU's speed, but not the only one — modern CPUs pipeline (overlap cycles) and run several instructions in parallel.

### Instruction Set Architecture (ISA)

The ISA is the contract between hardware and software: it defines which instructions exist, what they mean, and how they are encoded as bits.

- **x86-64** — the dominant ISA for desktops, laptops, and most cloud servers. Complex, with hundreds of instructions accumulated over decades (CISC).
- **ARM** — used in phones, tablets, and Apple Silicon Macs. Increasingly common in cloud servers (AWS Graviton). Simpler, fixed-width instructions (RISC). Generally more power-efficient.

x86 was designed in 1978 and has been adding instructions ever since while keeping backward compatibility. Its decoder must recognize hundreds of variable-length instructions (1 to 15 bytes each) and several legacy modes. That decoder costs transistors, and transistors cost power on every cycle.

ARM was designed in 1985 as a clean RISC architecture: fewer instructions, all 32 bits wide, a much simpler decoder. Less work per instruction means less power per instruction. That difference is why ARM wins wherever power matters more than compatibility:

- **Phones and tablets** — a power-hungry CPU drains the battery in hours. ARM had a head start on perf-per-watt and never lost it.
- **Apple Silicon Macs** (ARM, 2020+) — 20-hour battery life while matching Intel's raw performance.
- **Cloud** (AWS Graviton) — ~20% cheaper than equivalent x86 instances because the chips draw less power, and datacenters pay the electricity bill.

x86 still dominates desktops and most servers because of compatibility: decades of software, drivers, and operating systems are built for it. ARM is winning the new contexts (mobile, low-power laptops) and steadily chipping away at the old ones.

Programs compiled for one ISA do not run on another. When you run an x86 Docker container on an ARM machine or an AWS Graviton instance, the CPU or a software layer has to translate — which is why architecture matters when choosing cloud instance types or building container images.

### Pipelining and Out-of-Order Execution

Modern CPUs do not run one instruction at a time in strict sequence. They use several techniques to run faster:

- **Pipelining** — while one instruction is executing, the next is already being decoded, and the one after that is being fetched. Like an assembly line: multiple instructions in flight simultaneously.
- **Out-of-order execution** — if an instruction is stalled waiting for data (e.g., a cache miss), the CPU scans ahead and runs later instructions that are already ready. Results are committed in the original order.
- **Branch prediction** — at a conditional jump, the CPU guesses which branch will be taken and starts executing it before the condition is resolved. A wrong guess flushes the pipeline and costs ~15 cycles.
